---
title: "J-Expr syntax reference"
description: "J-Expr is the surface syntax for HashQL — JSONC extended with path expressions, call arrays, and typed data objects."
---

**J-Expr** (JSON Expression Language) is a small, machine-friendly surface syntax that maps one-to-one onto the HashQL AST. It is JSONC (JSON with comments) with three additions:

1. **Path expressions** (strings) — a tiny embedded DSL for absolute/relative paths, generics, and constraints.
2. **Call expressions** (arrays) — positional and labelled function arguments.
3. **Data expressions** (objects) — `#`-prefixed constructors for literals and composite values, with optional inline type refinement.

Every JSON type carries a distinct meaning in J-Expr:

| JSON type | J-Expr meaning |
| --------- | -------------- |
| String | Path or identifier |
| Array | Function call |
| Object | Data constructor (with `#` keys) |

## Forms at a glance

### Underscore

The JSON string `"_"` denotes a discarded value or an inference **hole**. Use it wherever you want the compiler to infer a type, skip a generic bound, or indicate a same-name import.

<CodeGroup>
```json Inference hole in return type
["fn", {"#tuple": []}, {"#struct": {"x": "Int"}}, "_", ["body"]]
```

```json No bounds for a generic
["fn", {"#struct": {"A": "_"}}, {"#struct": {"name": "A"}}, "Integer", ["body"]]
```

```json Same-name import
["use", "module", {"#struct": {"name": "_"}}, "body"]
```
</CodeGroup>

### Path expressions

Strings are parsed as paths or identifiers. Paths carry optional generic arguments and constraints using an embedded DSL.

- **Absolute** paths start with `::` and anchor to a package root.
- **Relative** paths resolve lexically from the current scope.

<CodeGroup>
```jsonc Absolute path with generics
"::core::math::add<Number, Integer>"
```

```jsonc Relative path with generic bounds
"Foo<T, U: Integer | String>"
```

```jsonc Simple variable
"x"
```

```jsonc Dotted path access sugar
"vertex.id.entity_id"
```

```jsonc Namespaced path
"::core::types::String"
```
</CodeGroup>

<Note>
  Absolute paths start at a package root (e.g., `::kernel::special_form::if`). Relative paths resolve lexically; `use` supports wildcards, selective imports, and aliasing. Locals follow standard shadowing rules; `let` can shadow an imported binding.
</Note>

### Call expressions

Arrays represent function calls: `[callee, arg1, arg2, ...]`. The first element is always the callee (a path string or nested expression); remaining elements are arguments.

**Labelled arguments** use a leading-colon key inside an object. Labels are auxiliary — positional order still matters.

<CodeGroup>
```jsonc Positional call
["::core::math::add", "x", "y"]
```

```jsonc Labelled arguments
["f", {":x": "x"}, {":y": "y"}]
```

```jsonc Mixed positional and labelled
["foo", {":a": "x"}, {":b": "y"}, ":c"]
```

```jsonc Graph query call
["::graph::head::entities", ["::graph::tmp::decision_time_now"]]
```
</CodeGroup>

### Data constructors

Objects with `#`-prefixed keys construct typed data values.

#### Literals

<CodeGroup>
```json Integer
{"#literal": 123}
```

```json Number (float)
{"#literal": 123.0}
```

```json String
"foo"
```

```json Boolean
true
```

```json Null
null
```
</CodeGroup>

#### Composite types

<CodeGroup>
```json Closed struct (invariant field set)
{"#struct": {"a": "a", "b": "b"}}
```

```json Open struct (width-subtyping allowed)
{"#struct(open)": {"name": {"#literal": "Alice"}}}
```

```json Tuple (fixed-length, heterogeneous)
{"#tuple": [{"#literal": 1}, {"#literal": "text"}]}
```

```json Dict (object syntax)
{"#dict": {"key1": "value1", "key2": "value2"}}
```

```json Dict (pair syntax)
{"#dict": [["key1", "value1"], ["key2", "value2"]]}
```

```json List (homogeneous sequence)
{"#list": [{"#literal": 1}, {"#literal": 2}]}
```
</CodeGroup>

#### Inline type refinement

Any data object may include a `"#type"` key to explicitly refine its type. This desugars to a call to the `as` special form.

<CodeGroup>
```jsonc Struct with inline type annotation
{
  "#struct": {"a": "a", "b": "b"},
  "#type": "(a: Integer, b: _)"
}
```

```jsonc Literal with explicit type
{"#literal": 3.14, "#type": "Float"}
```

```jsonc Struct with named type
{"#struct": {"name": {"#literal": "Alice"}}, "#type": "Person"}
```
</CodeGroup>

### Access sugar

J-Expr supports top-level access sugar that the frontend expands before further processing:

| Sugar | Desugars to | Meaning |
| ----- | ----------- | ------- |
| `a.b` | `[".", "a", "b"]` | Struct field access |
| `a.2` | `[".", "a", {"#literal": 2}]` | Tuple position access |
| `a[k]` | `["[]", "a", "k"]` | Dict/list index (returns `Option`) |

## Special forms

Special forms are **compiler-recognised macros** that lower to dedicated AST nodes. They are **not values** (cannot be stored or passed as arguments) and have **fixed arity**. Human-facing sugar desugars to a special form.

<AccordionGroup>
  <Accordion title="if — conditional">
    Evaluates `test`; if `true`, returns `then`. The two-argument form produces `Option<T>`; the three-argument form is total and returns `T` directly.

    ```text
    if/2<T>(test: Expr<Boolean>, then: Expr<T>) -> Option<T>
    if/3<T>(test: Expr<Boolean>, then: Expr<T>, else: Expr<T>) -> T
    ```

    <CodeGroup>
    ```jsonc Two-argument form (returns Option<T>)
    ["if", [">", "x", {"#literal": 0}], {"#literal": "positive"}]
    ```

    ```jsonc Three-argument form (total, returns T)
    ["if", [">", "x", {"#literal": 0}],
      {"#literal": "positive"},
      {"#literal": "non-positive"}]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="as — type ascription">
    Compile-time type assertion. Only **upcasts** are allowed (`T <: U`); downcasts would require runtime checks. Operationally inert — erased after type checking.

    ```text
    as/2<T, U>(value: Expr<T>, type: Type<U>) -> Expr<U>   where T <: U
    ```

    <CodeGroup>
    ```jsonc Upcast to a named type
    ["as", "value", "String"]
    ```

    ```jsonc Upcast with struct type
    ["as", ["get", "data", "field"], {"#type": {"name": "String", "age": "Int"}}]
    ```
    </CodeGroup>

    The `"#type"` key on a data object is syntactic sugar that desugars to an `as` call:

    ```jsonc
    {
      "#struct": {"a": "a", "b": "b"},
      "#type": "(a: Integer, b: _)"
    }
    ```
  </Accordion>

  <Accordion title="let — lexical binding">
    Binds `value` to `name` within `body`. Bindings participate in standard shadowing. A four-argument form accepts an explicit type annotation; the bound value must satisfy `U <: T`.

    ```text
    let/3<T, U>(name: Ident, value: Expr<T>, body: Expr<U>) -> Expr<U>
    let/4<T, U, V>(name: Ident, type: Type<T>, value: Expr<U>, body: Expr<V>) -> Expr<V>  where U <: T
    ```

    <CodeGroup>
    ```jsonc Simple binding
    ["let", "x", {"#literal": 42}, ["add", "x", {"#literal": 5}]]
    ```

    ```jsonc With explicit type annotation
    ["let", "x", "Int", {"#literal": 42}, ["add", "x", {"#literal": 5}]]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="type — type alias">
    Introduces a **type alias** (possibly generic with bounds) visible within `body`. Aliases use structural typing — two types are equivalent if they have the same structure, regardless of name.

    ```text
    type/3<T, U>(name: GenericIdent, value: Type<T>, body: Expr<U>) -> Expr<U>
    ```

    <CodeGroup>
    ```jsonc Simple alias
    ["type", "UserId", "String", ["body"]]
    ```

    ```jsonc Struct alias
    ["type", "Point", {"#type": {"x": "Float", "y": "Float"}}, ["body"]]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="newtype — nominal wrapper">
    Declares a **nominal wrapper** over a structural type (opaque identity). Useful for domain types (`Url`, `Uuid`) and for tagging constructors used in sum types. A constructor function of the same name is generated automatically and brought into scope.

    ```text
    newtype/3<T, U>(name: GenericIdent, value: Type<T>, body: Expr<U>) -> Expr<U>
    ```

    <CodeGroup>
    ```jsonc Domain type wrapper
    ["newtype", "UserId", "String", ["body"]]
    ```

    ```jsonc Construct the newtype
    ["newtype", "AccountId", "String",
      ["AccountId", {"#literal": "1234"}]
    ]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="use — imports">
    Resolves items under `path` into the local scope of `body`. Supports three import styles: wildcard, list, and selective (with aliasing).

    ```text
    use/3<T>(path: Path, items: Ident<*>,             body: Expr<T>) -> Expr<T>   // wildcard
    use/3<T>(path: Path, items: List<Ident>,           body: Expr<T>) -> Expr<T>   // list
    use/3<T>(path: Path, items: Dict<Ident, Ident|_>, body: Expr<T>) -> Expr<T>   // selective + aliasing
    ```

    <CodeGroup>
    ```jsonc Wildcard import
    ["use", "path::to::module", "*", ["body"]]
    ```

    ```jsonc List import (same-name)
    ["use", "path::to::module", {"#tuple": ["item1", "original"]}, ["body"]]
    ```

    ```jsonc Selective import with aliasing
    ["use", "path::to::module",
      {"#struct": {"item1": "_", "original": "renamed"}},
      ["body"]]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="fn — closure builder">
    Builds a **closure**. Generics may be annotated with bounds or left as inference holes (`_`). Parameters and return type are explicit; call sites leverage inference.

    ```text
    fn/4<T>(generics: Dict<Ident, _ | Type> | List<Ident>,
            parameters: Dict<Ident, Type>,
            returns: Type<T>,
            body: Expr<T>) -> Callable
    ```

    <CodeGroup>
    ```jsonc Simple closure (no generics)
    ["fn", [], {"#struct": {"x": "Int"}}, "Int", ["*", "x", {"#literal": 2}]]
    ```

    ```jsonc Generic closure (list-style generics)
    ["fn", {"#tuple": ["T"]}, {"#struct": {"x": "T", "y": "T"}}, "T", ["*", "x", "y"]]
    ```

    ```jsonc Generic closure (bounded generics)
    ["fn", {"#struct": {"T": "Int"}}, {"#struct": {"x": "T", "y": "T"}}, "_", ["*", "x", "y"]]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="input — host inputs">
    Introduces **immutable host inputs** (e.g., user-supplied parameters). Referential transparency is preserved: the input set is fixed at program start. An optional default value can be provided.

    ```text
    input/2<T>(name: Ident, type: Type<T>) -> T
    input/3<T>(name: Ident, type: Type<T>, default: T) -> T
    ```

    <CodeGroup>
    ```jsonc Required input
    ["input", "limit", "Int"]
    ```

    ```jsonc Input with default value
    ["input", "limit", "Int", {"#literal": 10}]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="access — static field/position selection">
    Static field or position selection. The selector must be a **literal** (an identifier for structs; a natural number for tuples). Unknown fields or indices are compile-time errors. Shorthand: `[".", value, field]`.

    ```text
    access/2<T: Struct, U>(value: Expr<T>, field: Ident)   -> T.field
    access/2<T: Tuple,  U>(value: Expr<T>, field: Natural) -> T.field
    ```

    <CodeGroup>
    ```jsonc Struct field access (explicit)
    ["access", "user", "name"]
    ```

    ```jsonc Struct field access (sugar)
    [".", "user", "name"]
    ```

    ```jsonc Tuple position access (sugar)
    [".", "pair", {"#literal": 2}]
    ```
    </CodeGroup>
  </Accordion>

  <Accordion title="index — dynamic lookup">
    Dynamic lookup in dictionaries or lists. Returns `Option<V>` to represent the possibility of a missing key. Shorthand: `["[]", value, index]`.

    ```text
    index/2<K, V>(value: Expr<Dict<K, V>>, index: Expr<K>)       -> Option<V>
    index/2<V>(   value: Expr<List<V>>,    index: Expr<Integer>) -> Option<V>
    ```

    <CodeGroup>
    ```jsonc Dict lookup (explicit)
    ["index", "myDict", "myKey"]
    ```

    ```jsonc Dict lookup (sugar)
    ["[]", "myDict", "myKey"]
    ```

    ```jsonc List index (sugar)
    ["[]", "items", {"#literal": 0}]
    ```
    </CodeGroup>
  </Accordion>
</AccordionGroup>

## Full example

The following query fetches all non-archived entities using the graph package, demonstrating `let`, labelled calls, `fn`, and path expressions together:

```jsonc
["let", "entities",
  ["::graph::head::entities", ["::graph::tmp::decision_time_now"]],
  ["filter", "entities",
    ["fn", {"#tuple": []}, {"#struct": {"e": "_"}}, "_",
      ["==", "e.archived", {"#literal": false}]]]]
```

## Appendix: J-Expr cheat sheet

**Access sugar**

```text
a.b   →  [".", "a", "b"]
a.2   →  [".", "a", {"#literal": 2}]
a[k]  →  ["[]", "a", "k"]
```

**Calls**

```jsonc
["::core::math::add", "x", "y"]
["f", {":lhs": "x"}, {":rhs": "y"}]
```

**Data constructors**

```jsonc
{"#literal": 123}
{"#tuple": [...]}
{"#struct": {...}}
{"#struct(open)": {"a": 1}}
{"#dict": {...}}
{"#dict": [[k, v], ...]}
{"#list": [...]}
```

**Paths**

```jsonc
"::core::math::add<Number, Integer>"
"math::add"
"Foo<T, U: Integer | String>"
```
