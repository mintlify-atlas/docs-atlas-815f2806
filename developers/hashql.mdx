**HashQL** (HASH Query Language) is the native query language for the HASH graph. It is a typed, functional traversal language that operates over bi-temporal, strongly typed graphs. Its surface syntax is intentionally machine-friendly and syntax-agnostic: multiple frontends can emit into a single canonical AST with one unified semantics.

The library lives at `libs/@local/hashql/`.

## Key Characteristics

| Property | Description |
|----------|-------------|
| **Functional traversal core** | Queries are compositions of pure operators over graph effects |
| **Sound, expressive types** | Structural types by default; nominal opt-in via `newtype`; bounded generics; unions/intersections; μ-types; HM-style inference with subtyping |
| **Syntax independence** | Current frontend is J-Expr; additional frontends can be added without disturbing downstream phases |
| **First-class diagnostics** | Every phase emits structured diagnostics; fatal issues stop at diagnostic boundaries while non-fatal ones accumulate |
| **Heterogeneous execution** | Targets multiple backends (PostgreSQL, planned interpreter); cost-aware placement |
| **Bi-temporal graphs** | Queries operate over a bi-temporal slice parameterised by decision time and transaction time |

## Compiler Pipeline

HashQL's compiler is organised into four phases. Only the frontend **pushes** into its successor; all other phases **pull** from their single predecessor.

```
Frontend (J-Expr)
       │
       ▼
Abstract Syntax Tree (AST)
  PreExpansionNameResolver
  → SpecialFormExpander
  → Sanitizer
  → ImportResolver
  → NameMangler
  → TypeDefinitionExtractor
  → NodeRenumberer
  → TypeExtractor
       │
       ▼
Higher-level Intermediate Representation (HIR)
  Reification
  → AliasReplacement
  → ConvertTypeConstructor
  → TypeInference
  → TypeChecking
  → Specialization
       │
       ▼
Evaluation (currently: PostgreSQL)
```

**Phase responsibilities**

- **Frontend (J-Expr):** Streaming lexer and compact parser that reads JSONC and produces the AST directly — no intermediate JSON DOM is materialised.
- **AST:** Expands special forms, canonicalises and erases surface type syntax, resolves imports and names, and assigns stable IDs via small, testable passes.
- **HIR:** Reified, interned tree for type inference, type checking, and specialisation. Retains an explicit tree to model control flow and typing constraints.
- **Evaluation:** Currently lowers to a transient graph IR executed by the PostgreSQL backend. A richer interpreter and a MIR (Middle-level IR) with basic blocks are planned.

<Note>
  A **MIR** layer with basic blocks is planned to unlock data- and control-flow optimisations and parallel scheduling. An **interpreter** is also planned to execute HashQL programs that cannot be pushed down to the database.
</Note>

## J-Expr Syntax

**J-Expr** is HashQL's current surface syntax — a small, machine-friendly format that maps one-to-one onto the AST. It is JSONC (JSON with comments) extended with three additions:

1. **Path expressions** — strings with an embedded DSL for absolute/relative paths, generics, and constraints.
2. **Call expressions** — arrays where the first element is the callee.
3. **Data expressions** — objects for literals and composite values, with optional inline type refinement.

### Core forms

| Form | Syntax | Meaning |
|------|--------|---------|
| Underscore | `"_"` | Discarded value or inference hole |
| Absolute path | `"::core::math::add<Number, Integer>"` | Fully-qualified name |
| Relative path | `"math::add"` | Lexically resolved name |
| Call | `["callee", arg1, arg2]` | Function application |
| Labelled call | `["f", {":x": "x"}, {":y": "y"}]` | Keyword arguments (positional order still matters) |
| Integer literal | `{"#literal": 123}` | Integer |
| Float literal | `{"#literal": 123.0}` | Number |
| Struct (closed) | `{"#struct": {"a": 1, "b": 2}}` | Closed record |
| Struct (open) | `{"#struct(open)": {"a": 1}}` | Open (width-subtyped) record |
| Tuple | `{"#tuple": [x, y, z]}` | Heterogeneous fixed-length tuple |
| Dict | `{"#dict": {"k": v}}` | Homogeneous key-value map |
| List | `{"#list": [v1, v2]}` | Homogeneous list |

### Access sugar (embedded DSL)

```text
a.b    ≡  [".", "a", "b"]              // field access
a.2    ≡  [".", "a", {"#literal": 2}]  // tuple element access
a[b]   ≡  ["[]", "a", "b"]            // dict/list index (returns Option)
```

## Special Forms

Special forms are compiler-recognised macros that lower to dedicated AST nodes. They are not values (cannot be stored or passed as arguments) and have fixed arity.

### `if` — conditional expression

```text
if/2<T>(test: Expr<Boolean>, then: Expr<T>) -> Option<T>
if/3<T>(test: Expr<Boolean>, then: Expr<T>, else: Expr<T>) -> T
```

The two-argument form produces `Option<T>`; the three-argument form is total and returns `T`.

### `as` — type ascription

```text
as/2<T, U>(value: Expr<T>, type: Type<U>) -> Expr<U>   where T <: U
```

Only **upcasts** are allowed. The form is operationally inert and erased after type checking.

### `let` — lexical binding

```text
let/3<T, U>(name: Ident, value: Expr<T>, body: Expr<U>) -> Expr<U>
let/4<T, U, V>(name: Ident, type: Type<T>, value: Expr<U>, body: Expr<V>) -> Expr<V>  where U <: T
```

### `type` — type alias

```text
type/3<T, U>(name: GenericIdent, value: Type<T>, body: Expr<U>) -> Expr<U>
```

Introduces a type alias (possibly generic with bounds) visible within `body`.

### `newtype` — nominal wrapper

```text
newtype/3<T, U>(name: GenericIdent, value: Type<T>, body: Expr<U>) -> Expr<U>
```

Declares a nominal wrapper over a structural type (opaque identity). Used for domain types (`Url`, `Uuid`) and for tagging constructors in sum types.

### `use` — import

```text
use/3<T>(path: Path, items: Ident<*>, body: Expr<T>) -> Expr<T>           // wildcard
use/3<T>(path: Path, items: List<Ident>, body: Expr<T>) -> Expr<T>        // list
use/3<T>(path: Path, items: Dict<Ident, Ident|_>, body: Expr<T>) -> Expr<T>  // selective + aliasing
```

### `fn` — closure

```text
fn/4<T>(
  generics: Dict<Ident, _ | Type> | List<Ident>,
  parameters: Dict<Ident, Type>,
  returns: Type<T>,
  body: Expr<T>
) -> Callable
```

### `input` — host input

```text
input/2<T>(name: Ident, type: Type<T>) -> T
input/3<T>(name: Ident, type: Type<T>, default: T) -> T
```

Introduces immutable host inputs (e.g., user-supplied parameters). Referential transparency is preserved: the input set is fixed at program start.

### `access` — static field selection

```text
access/2<T: Struct, U>(value: Expr<T>, field: Ident)    -> T.field
access/2<T: Tuple, U>(value: Expr<T>, field: Natural)   -> T.field
```

Shorthand sugar: `[".", value, field]`. Unknown fields/indices are compile-time errors.

### `index` — dynamic lookup

```text
index/2<K, V>(value: Expr<Dict<K, V>>, index: Expr<K>)       -> Option<V>
index/2<V>(value: Expr<List<V>>, index: Expr<Integer>)        -> Option<V>
```

Shorthand sugar: `["[]", value, index]`.

## Type System

HashQL's type system is designed to be compatible with graph schemas and expressive enough for a modern functional language.

### Core model

- **Structural by default**, with **nominal opt-in** via `newtype` (opaque identity).
- **Complete lattice** with `⊤` ("Unknown") and `⊥` ("Never").
- **Unions** (`T | U`) and **intersections** (`T & U`) are supported and distributive.
- **Bounded parametric polymorphism** with generic constraints (`T: Number | String`).
- **Equi-recursive μ-types** for cyclic structures.
- **Variance:** covariant by default; closure parameters are contravariant.
- **Kinds** separate types (`*`) from higher-order type constructors (`* → *`).

### Composite types

| Type | Description |
|------|-------------|
| **Struct (closed)** | Invariant field set; fields are covariant |
| **Struct (open)** | Width-subtyping; additional fields allowed |
| **Tuple** | Fixed-length, heterogeneous; length is invariant |
| `Dict<K, V>` | Homogeneous key/value; keys invariant, values covariant |
| `List<V>` | Homogeneous; covariant in element type |

### Enumerations via `newtype`

Algebraic data types are encoded as unions of nominal wrappers over structural carriers:

```text
Option<T>  ≜  Some<T> | None
Some<T>    ≜  newtype T
None       ≜  newtype Unit   (Unit is the empty tuple ())

Result<T, E>  ≜  Ok<T> | Err<E>
Ok<T>         ≜  newtype T
Err<E>        ≜  newtype E
```

### Inference and checking

- Constraint-based inference (HM-style with subtyping) over the HIR: emit constraints, solve, then apply a substitution bottom-up.
- **Inference holes** (`_`) must be constrained by usage.
- **Inference variables** (`T`) may remain unconstrained if latent in generics.
- Bidirectional checks around `as`, parameter types, and return types improve local precision and error pointing.

## Packages and Modules

HashQL ships with four built-in packages:

| Package | Role |
|---------|------|
| `kernel` | Type primitives and special forms (compiler-intrinsic) |
| `core` | Foundational types and constructors: arithmetic, comparison, options, results |
| `graph` | Graph-specific effect types and intrinsics arranged as **head → body → tail** |
| `main` | Implicit user entry point |

The `graph` package follows a three-part structure for query construction:

- **Head** — query entry points: pick starting vertices/edges, choose temporal axis and range.
- **Body** — compositional traversal operators: map, filter, expand, temporal joins, hops.
- **Tail** — terminals: materialise, aggregate, paginate, authorise.

User-defined modules are planned but not yet supported.

## Graph Effects and Execution

Compilation may **push down** eligible query fragments to PostgreSQL while keeping non-pushable fragments in an interpreter, using cost-aware placement and parallelism where safe. The permission layer can be injected as guards during pushdown or as an overlay during interpretation.
